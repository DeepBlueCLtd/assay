# ASSAY — Decision Register

**Assessment Semantics & Scenario Analysis**
Status: canonical · sole home of decisions · opened 2026-07-11
Regime: register-first (ASSAY-DEC-2). A decision is recorded here before it is cited or relied upon in any peer document, spec, or plan. Peer documents are updated by batch propagation in the same change that lands the entry. Methodology never originates in the build plan; spec-kit artefacts (including the constitution) are projections of this register.

Entry conventions: IDs are `ASSAY-DEC-n`, allocated once, never reused. An entry is superseded by a later entry, never edited beyond typographical repair. Candidates awaiting ratification live in `assay-concept.md` §6 and `assay-ui-design.md` §6, not here.

## Batch 1 — Founding design session (2026-07-11)

Ratified in the founding design session and carried in the scaffold until this register was split out (the split is the batch-2 session's first act; see ASSAY-DEC-12 rationale). Entries reproduced verbatim, with one pointer repaired: DEC-11's "§7 below" now reads `assay-concept.md` §7, since "below" no longer holds outside the source document.

| ID | Decision | Rationale |
|---|---|---|
| ASSAY-DEC-1 | **ASSAY established** as a standalone demonstrator: JIPOE-derived knowledge, quantified and typed, exploited by optimisation machinery; one environment, multiple explorable theses (A–H), multiple demo narratives over a shared backend. | A single coherent demonstration beats discrete thesis builds; the theses share ~80% of their machinery. |
| ASSAY-DEC-2 | **Full REMIT-style documentation discipline**: own register with own DEC numbering; register-first (decisions recorded before citation in peer documents); batch propagation across the canonical set; methodology never originates in the build plan. | The discipline proved its worth in REMIT; a demonstrator that reconsiders principles must record its reconsiderations. |
| ASSAY-DEC-3 | **Fully independent LinkML schema.** No import of REMIT shapes; REMIT concepts re-derived only where they earn their place. Convergence/divergence findings reported back as REMIT register candidates. | The point is re-derivation; a dependency would prejudge the answer. |
| ASSAY-DEC-4 | **In-browser mock implementation behind a service seam contract** (TS/TSX, REMIT v1 pattern): fully client-side and offline; REST-shaped contracts mature early; real services can swap in later with no client change. **Invariant: the scorer is honestly real even in the mock** — sensitivity, discrimination, and robustness demos are re-scoring loops and become theatre over a fake scorer. | Fastest demo loop; the seam is the durable asset. |
| ASSAY-DEC-5 | **Architecture**: one content-addressed store; compile / plan-score / analysis services behind the seam; the **traceability graph as a first-class object** (knowledge → channel → verdict → rationale edges); role surfaces as config-declared projections over the shared store; every cross-surface write a stamped delta. A surface may only *arrange* existing projections, never demand new backend semantics. | Every narrative is a read of the trace graph from a different end; surfaces stay cheap only if the graph is real. |
| ASSAY-DEC-6 | **The knowledge object is the central new type**: a quantified JIPOE Q&A carrying encoding class (hard constraint / banded soft cost / scenario weight), banded value, provenance (source class, confidence, ownership), criticality, and validity window. Detailed in `assay-knowledge-model.md`. | The one genuinely new schema relative to REMIT's model; everything else is a familiar shape re-derived. |
| ASSAY-DEC-7 | **Theses are explorations, not builds**; narratives are demo configurations. No thesis-specific backend forks. | Prevents the demonstrator fragmenting into four half-products. |
| ASSAY-DEC-8 | **Vignette**: deliberately fictional archipelago setting; small joint task force echelon (3–5 force elements, red ORBAT with 2–3 genuinely divergent COAs, one operational objective). The vignette is engineered by construction to exercise every narrative: conflicting commitments (B), a load-bearing deceivable assessment (E), divergent red COAs (C), a collectable discriminator (D), a perishable forecast (F). Detailed in `assay-vignette.md`. | Fiction removes classification-adjacent realism constraints; JTF echelon gives JIPOE texture without ORBAT sprawl. |
| ASSAY-DEC-9 | **Banded-honesty invariant**: no scalar derived from an *assessed* source reaches any surface unbanded; provenance and an "assessment, not fact" marking are visible wherever such a number is shown; plan dominance is judged in banded space only. | One SME seeing a confident number they know is guesswork discredits the whole pipeline. Inherits the Blockbuster lesson and REMIT NF9/NF10 by re-derivation, not import. |
| ASSAY-DEC-10 | **Scoring factored as an independently callable unit** of the planning service. Analysis services (sensitivity, discrimination, staleness) are thin orchestrations: the scorer called in a loop plus a trace-graph walk — never separate engines. | Keeps theses C/D/E/F nearly free once A and B stand. |
| ASSAY-DEC-11 | **Research-first development discipline.** Every build stage opens with a bounded research task on the doctrine and literature that stage touches (JIPOE/IPOE and military planning doctrine; optimisation techniques). Findings are recorded as a short research note in `docs/research/` *before* implementation begins; notes cite sources and state what the stage will do differently because of them. Doctrinal or algorithmic claims in canonical documents must trace to a research note or a register entry, not to developer recollection. The build plan (`assay-build-plan.md`) names each stage's research prompts; `assay-concept.md` §7 lists starting points. | ASSAY's credibility rests on doctrinal and mathematical honesty. Developers who haven't read the doctrine will build a strawman JIPOE; developers who haven't read the optimisation literature will reinvent it badly. Bounded research (hours, not weeks) with a written note keeps this cheap, auditable, and cumulative. |

## Batch 2 — Canonical-set authoring session (2026-07-11)

Ratified in the structured interview that opened the document workstream (delivery plan §2.1, slices D1–D4). Interview answers are the rationale record; where an entry constrains the schema, `assay-knowledge-model.md` carries the normative elaboration.

| ID | Decision | Rationale |
|---|---|---|
| ASSAY-DEC-12 | **Document slices are register/authoring work, not spec-kit features.** The canonical documents (D1–D5) are authored directly as documents under register discipline; the spec-kit machinery (`specs/NNN-name/` → spec → plan → tasks) begins at SPEC-01 and applies only to build slices. | Spec-kit's shape — acceptance scenarios, testable exit criteria — fits code; a founding document's acceptance is register/design review. Specs must also trace *to* the canonical documents, so the documents cannot themselves be spec-kit products without circularity. Ratifies the delivery plan §2 rule. |
| ASSAY-DEC-13 | **`assay-architecture.md` (slice D5) is deferred** until the knowledge model and seam contract exist. In the interim, orientation lives in `assay-concept.md` §3 and the invariants G1–G5 live normatively in `assay-seam-contract.md` §G (projected into the constitution). | D5 sits off the critical path (delivery plan §3); writing it before D2/D3 would speculate about the very shapes it must document. Deferred explicitly, by decision, not omission. |
| ASSAY-DEC-14 | **Fact/assessment split and source-class enum.** Source classes are `observed / reported / assessed / assumption`. Only `observed` — directly charted, surveyed, or own-force data, never inferred — is *fact* and may render unbanded. `reported`, `assessed`, and `assumption` values are always banded and always carry the "assessment, not fact" marking (DEC-9). A commander's *directive* quantity (a commitment threshold) is fact-of-intent, not an assessment, and may be scalar. | Banding charted runway lengths would be absurdity that discredits the discipline it's meant to protect; the narrow definition of fact keeps the exemption from leaking. The four classes are generic rather than INT-discipline-flavoured, keeping the fiction unclassifiable and the schema domain-honest. |
| ASSAY-DEC-15 | **Bands are pure closed intervals** `{lo, hi, unit}`: no stored midpoint, no distributional claim, anywhere in the schema. Scoring propagates intervals (interval arithmetic / worst-and-best-case within band); dominance and verdicts are judged in banded space only. | A stored midpoint is an invitation to false precision — it *will* end up on a surface. Extends DEC-9 from rendering into representation. |
| ASSAY-DEC-16 | **ConfidenceBand enum is ICD 203's `low / moderate / high`**, fixed in the schema now; the numeric confidence → band-width mapping is deferred to the Stage-1 research note (`01-knowledge.md`) and enters the schema as an annotation when that note lands. | The enum is stable doctrine and unblocks the schema; inventing the width mapping without the ICD 203 reading is precisely the failure DEC-11 exists to prevent (build plan §risks). |
| ASSAY-DEC-17 | **The JIPOE question is embedded in the KnowledgeObject**: an object may exist as an open, unanswered question (what the S1 collection queue ranks). Lifecycle: `open → answered → (superseded | stale | contested) → resolved | retired`. Staleness is driven by the validity window measured against the **scenario clock** (D+n), never wall clock. | One type carries the Q&A pair through its whole life, so exposure, staleness, and collection queues are views of one lifecycle rather than joins across types. Wall-clock staleness would break stamp determinism (constitution IV / seam G1) and make offline demos non-reproducible. |
| ASSAY-DEC-18 | **Open questions carry expected-answer-per-COA links** — a miniature event matrix: for each ScenarioCOA, the band the answer is expected to fall in if that COA is the truth. Discrimination (`/analyse/discrimination`, thesis D) is computed from the separation of these expected bands, never asserted. | The doctrinally honest analogue of the JIPOE event matrix (indicators discriminate between adversary COAs); SPEC-12 is unbuildable without it. |
| ASSAY-DEC-19 | **Commitments are ordered by commander-owned ordinal tiers** `must / should / prefer`. No numeric commitment weights anywhere. Within-tier ties in relaxation are broken explicitly and the tie-break is stated in the relaxation report — never silently. | Numeric sacrifice weights would be false precision exactly where SMEs will look hardest (the least-worst argument). Ordinal tiers match how guidance is actually issued; the visible tie-break keeps the honesty when ordinality runs out. |
| ASSAY-DEC-20 | **A Plan is per-force-element timed routes plus task windows** on the grid world — nothing richer in v1. No inter-element dependency links, no abstract action language. | Everything the scorer and verdict matrix need, nothing the generator can't produce; richer plan algebra is horizon work that would stall the spine. |
| ASSAY-DEC-21 | **Two-tier identity**: every object carries a stable logical id (e.g. `K9`, `C2`) and each version is content-addressed by hash; revision is supersession recorded as a trace edge between versions (or between distinct logical objects where a new object supersedes an old one, e.g. K9 superseding K5). Mutation does not exist. | Content addressing gives determinism and tamper-evidence (DEC-5, constitution IV); the logical id gives humans and fixtures a stable handle across versions. Supersession-as-edge makes staleness a trace walk, which is what thesis F demonstrates. |

| ID | Decision | Rationale |
|---|---|---|
| ASSAY-DEC-22 | **`assay-walkthrough.md` joins the canonical set** as the standing end-to-end validation of interactions, structures, and flows. Its §0 discipline rule now binds: a step the seam contract cannot perform is a *contract* defect — fixed in the contract and logged in walkthrough §9, never worked around in the walkthrough — and the document is re-walked whenever the contract or knowledge model changes. Closes concept §6.6. | The three things ASSAY demonstrates — user interactions, data structures, data flows — are each held by a canonical document but played through by none; the walkthrough is the only artefact that exercises them together on a real heartbeat, so it earns canonical standing and the contract-defect rule that keeps the seam honest against actual use. |
| ASSAY-DEC-23 | **Per-stage demo moments are a standing exit requirement.** Every build-plan stage exit is paired with a *user-observable twin* — what a watcher sees on a surface when the criterion holds — and a scripted **two-minute demonstration** drawn from the vignette coverage matrix and shown from the running build. The Stage-7 narratives *assemble* these banked moments; they do not replace them. Closes concept §6.7. | Machine-observable exit criteria alone let a stage "pass" with nothing a person can see; pairing each exit with an observable twin and a scripted demo keeps the build accountable to its audience at every stage and banks the narrative material as it is earned. Presentation of the exits the vignette already engineers (DEC-7/DEC-8), not new methodology. |
| ASSAY-DEC-24 | **Seam contract v0.2 additions are ratified register content**: object listing `GET /objects?class=` (seam §2), delta publication on *every* knowledge write — create / supersede / contest / resolve (seam §3), and the select service `POST /select` (seam §11). Adopted to close the four gaps the end-to-end walkthrough surfaced (walkthrough §9); previously draft-document content resting on DEC-5 by implication. Closes concept §6.8. | The walkthrough found real gaps — plan/rationale candidates with no serving endpoint, cross-surface knowledge writes that published no delta, and the commander's decisive act as the one write in the system with no service behind it (a dead-end backward chain, violating G3). These are seam semantics, not surface arrangement, so they belong in the register under their own entry rather than leaning on DEC-5's general authority. |
| ASSAY-DEC-25 | **G6 — propagation honesty — is a ratified seam invariant**, standing alongside G1–G5: widening any input band never narrows any output band, and every point-realisation of the inputs yields a result inside the output band. This is G2 stated mathematically rather than typographically. Hand-computed instances are the vignette §9 oracle cases (O-1…O-4), binding on SPEC-07 acceptance and re-asserted end-to-end by the spine-complete gate (SPEC-15). Closes concept §6.9. | A scorer can satisfy G1–G5 while propagating intervals wrongly — precisely the false-precision machine ASSAY refuses — so the band arithmetic needs an acceptance criterion of its own, not just the typographic ban of G2. The oracle cases already test it from Stage 3; ratification makes what was already gating explicit and standing. |
| ASSAY-DEC-26 | **`assay-findings.md` joins the canonical set** as the sole home of findings: thesis states with transition criteria, re-derivation verdicts, and the DEC-3 REMIT-candidate channel. A finding may now be cited as authority for what an exploration *concluded* — never as the origin of a decision, since the ledger originates nothing (DEC-2 unbroken). Closes concept §6.10. | Findings are ASSAY's unique product ("where a principle survives re-derivation it is validated as general; where it doesn't, that is a finding"), yet until ratified they had no citable home and the re-derivation evidence would have been unrecoverable by the spine gate. Canonical standing gives conclusions a home without breaching the DEC-3 firewall — findings still reach REMIT only as candidates. |
| ASSAY-DEC-27 | **SME evaluation checkpoints (comms plan §12) are standing gates in the thesis-transition rules**: no thesis moves `explored → concluded` without at least one checkpoint having seen it, and every SME reaction bearing on a thesis or principle lands in the findings ledger within a day — verbatim where possible, and, where it challenges a decision, as a register candidate. Closes concept §6.11. | A demonstrator whose primary audience is doctrine SMEs cannot let a thesis conclude on its engineered success path alone; the checkpoint gate forces at least one encounter with real J-2/J-3 judgement before "concluded", and the capture rule makes that encounter cumulative evidence rather than anecdote. A session that produces no ledger entries is a session that wasn't listened to. |

## Batch 4 — Delegated-authority ratification (2026-07-14)

Ratifies the seven candidates held in `assay-concept.md` §6.12–6.18 — each built under DEC-2's delegated-authority escape hatch (flagged, not asserted) and now closing with a register entry. The pattern was established by items 6.12/6.13 (sparse channels, home-page currency) and extended to the front-end lane (items 16–18) and the flow-view behaviours (item 15). Peer documents swept in the same change: concept §6 items struck through; ui-design §6 items 1–2 resolved; constitution bumped to 1.2.0; delivery plan §2 taxonomy updated; CLAUDE.md status line updated.

| ID | Decision | Rationale |
|---|---|---|
| ASSAY-DEC-28 | **Home-page currency at spec granularity.** Each build specification carries, as part of its definition of "done," a home-page-currency step: assess whether the slice moves a thesis or stage from planned to demonstrated and, if so, update `docs/assay-home.html` and `docs/status.yml` accordingly — under the same banded-honesty guardrail the site inherits (nothing shown as done that the repository does not contain; a stage cannot show "building" before its research note is published). Progress reporting is part of building at *spec* granularity, extending the comms plan §9 ownership rule from stage to slice. Closes concept §6.12. | A stage gate that only fires at the end of seven stages means months of silence on the public site while work lands. Spec-granularity currency makes progress visible as it is earned — and the banded-honesty guardrail already prevents overclaiming, so the only cost of finer grain is the bookkeeping, which is cheaper than the communications debt it prevents. |
| ASSAY-DEC-29 | **Sparse channel representation.** A `Channel` is a `default: Band` plus a sparse set of named, optionally time-boxed `RegionOverride`s, with `RegionName → cell set` geometry held once in a `VignetteConfig` object and region→cell materialisation a lazy, unstored, score-time function. Dense `Channel.cells` is retired. This is a change to a canonical LinkML shape (knowledge model §7) and to the seam contract (open item 2, which pre-delegated the representation choice to the Stage-2 research note). Closes concept §6.13. | Research note `02-compile.md` demonstrates the necessity: the Stage-0 benchmark showed a dense `CompiledWorld` is ~1.2 M cells / ~84.9 MB / ~19.4 s serialise+hash — not viable even in the mock. ATP 2-01.3 MCOO doctrine records deviations (overlays), not per-cell attributes; sparse channels are therefore both doctrinally honest and computationally necessary. The pre-delegation in seam open item 2 anticipated exactly this outcome. |
| ASSAY-DEC-30 | **Communications artefacts are register-blessed authoring work, not spec-kit features.** Three slice categories now exist: document slices (D#), build specifications (SPEC-##), and communications artefacts (the Home page, the roadmap/updates feed, blog articles). Communications artefacts are projections of already-shipped work, authored under register/comms honesty discipline; they are never themselves spec-kit features. A blog article reports on a slice that already went through spec-kit; the article does not get its own `specify → plan → tasks` pass — it is part of the reporting owed at that slice's gate (comms §9, and the home-page-currency step of DEC-28). The build path is the hand-rolled static assembler (`scripts/build-site.ts`), not Jekyll — a change from a non-binding comms plan §8 recommendation, not a ratified decision. Closes concept §6.14. | DEC-12's reasoning extended from founding documents to communications: spec-kit's shape (testable acceptance scenarios) fits code, not a dated public claim, and a communications page must trace *to* the build, so it cannot be a spec-kit product without circularity. The static assembler keeps one build path and avoids introducing a dependency (Jekyll) for one output channel. |
| ASSAY-DEC-31 | **Flow-view behavioural decisions (a/b/c ratified; d deferred).** Three behaviours of the system-flow infographic (SPEC-14 flow-view sub-slice, research note `07-flow-view.md`) are ratified: **(a)** live auto-recompute in the flow view, attribution-visible — the fan-out fires automatically but remains stamped, delta-logged, and gate-policed (badge flips, a delta lands, the gate pulses); this is deliberately distinct from S2's "recompile when ready" (ui-design §3.4.4) but preserves the walkthrough's "nothing recomputes silently" because attribution is mandatory, only the trigger is automated; **(b)** bounded viewer-write sandbox palette (toggle R1/R2/R3/R3m, contest/resolve K12, supersede K5→K9, grant/withhold W-1) against the real mock seam — a surface-behaviour decision expanding S4 from read-only projection to bounded interaction; **(c)** the map/geospatial panel (ui-design §6.1) — a Meridian schematic rendered from real `VignetteConfig` region geometry with the current world's threat/mobility channel overrides overlaid live, a projection of the compiled world, never a hand-drawn claim. Behaviour **(d)** — the truly-silent "you-are-the-optimiser" teaching toggle — is **explicitly deferred** for an SME reaction (DEC-27 checkpoint). Closes concept §6.15; resolves ui-design §6.1. | The three ratified behaviours were all demonstrated on Meridian (SPEC-14 flow-view tests AS-1…AS-11) and each is a pure projection of existing seam behaviour — no new engine, no new compute, no new decision-making. The auto-recompute honesty distinction (attribution-visible trigger vs silent recomputation) was validated by the research note (`07-flow-view.md` §3: the protected property is attribution, not manual triggering, because the flow view is an *explainer*). Behaviour (d) is deferred honestly — a "you are the optimiser" toggle that suppresses the system's own computation is a value judgement about when silence is pedagogically useful, and DEC-27 requires that to be reacted to by an SME. |
| ASSAY-DEC-32 | **Surface shell: four literal role tabs in one single-page app over a shared client store.** The long-open shell question (concept §6 item 4, ui-design §6 item 2) is decided: four literal role tabs — J-2, J-3/5 planner, Commander, Observer — in one page, over one in-browser object+trace store and one instance of the real services. The "different roles" illusion tabs weaken is recovered honestly by the propagation glow (DEC-34): an edit on one tab visibly propagates to the others. Closes concept §6.16; resolves concept §6.4 and ui-design §6.2. | Research note `05-surfaces.md` evaluates the alternatives (routed micro-frontends, role-switch with distinct chrome) and finds them costlier to build and maintain with no honesty gain — the shell/pure-component split (DEC-33) keeps components extractable regardless of the hosting frame. Tabs make the cross-role propagation visible at a glance, which is a demonstrator asset: the heartbeat loop *is* the point, not chrome to hide it behind. |
| ASSAY-DEC-33 | **Live in-browser pipeline + editable surfaces via the shell/pure-component split.** The published app runs the actual Knowledge/Compile/Score/Handful/Relax services client-side (the hashing targets `globalThis.crypto.subtle` for one browser+Node code path), and operator edits are a first-class authoring act — the surfaces become the light authoring surface, seeded from fixtures (resolving concept §6 item 5's direction). The SPEC-14 extractability constraint is resolved, not waived, by a **shell / pure-component split**: the eight `src/components/*` renderers stay pure (types-in, HTML-string-out); all state, events, recompute, and propagation live in a new `src/app/` shell layer. Every edit is gated by the existing honesty machinery (`encoding.ts`, `lint.ts`, contested-knowledge refusal) — a dishonest edit returns a first-class `Refusal`/`waiver_required`, never a silent fabricated result, and no assessed scalar is rendered unbanded under any edit (G2). Closes concept §6.17. | Running the real pipeline client-side is the most honest option: the demonstrator claims to show a pipeline, and a mock would undermine that claim. The shell/pure-component split costs nothing structurally (it is the MVC separation the code already wanted) and preserves the SPEC-14 extractability constraint — components that depend only on generated types are reusable in REMIT or any other consumer, which was the delivery plan's stated durable asset. |
| ASSAY-DEC-34 | **Propagation feedback ("glow") is G6 made visible.** When an edit on one surface changes data another surface depends on, the affected tab and downstream components glow for a fixed window — derived, not decorative: a unit glows iff its displayed value changed on recompute (each pure component stamps `data-glow-id` + value `data-glow-sig`; the shell diffs by value), so the glow is row/cell-level and never over-reports a mere re-derivation. The glow is the standing propagation-honesty invariant (G6, DEC-25) rendered as an operator-visible affordance — it must never under-report any more than the scorer may silently drop a consequence. A companion one-hop "informs / influenced-by" context menu on any item exposes the trace graph on demand. Closes concept §6.18. | G6 was ratified in batch 3 (DEC-25) as a mathematical invariant; the glow is its rendering. The value-keyed mechanism (diff by `data-glow-sig`, not by recompute event) prevents both over-reporting (a re-derivation that produces the same value does not glow) and under-reporting (every changed downstream value glows, even if the edit was indirect). The trace-menu companion gives the "why" behind the glow without requiring the full transitive dependency view (tracked as horizon issue #24). |
