Design notes — option A

  • Reading direction: left-to-right matches the "informs → influences" mental model. The pipeline already flows this way in the system-flow infographic.
  • Fan-out: when a lane has >5 items, collapse to "show N → all" with the count visible. Never silent truncation (G3).
  • Scale: K8's downstream reaches ~20 verdicts across 3 scenarios × 6 commitments × 2 plans. The "+ N more" affordance handles this; clicking expands the lane.
  • Connector detail: edge-type labels on every connector. In implementation, these become hoverable for the full edge metadata.
  • Item detail: the focus node and its immediate neighbours show a brief annotation (band, status, scenario). Deeper nodes show id only — detail on hover.
  • Re-focus: clicking any node re-centres the view on it. The one-hop trace menu stays available as a right-click / long-press.

Recommended: the horizontal flow is natural for this domain, works at Meridian scale, and is the simplest to implement (CSS flex lanes, no canvas/SVG graph engine needed).

Design notes — option B

  • Familiar form: the vertical DAG is the standard dependency-graph idiom. Easier to scan when the upstream is wide (many knowledge items feeding one world).
  • Width pressure: at Meridian scale, the verdict tier (12 items: 2 plans × 6 commitments) wraps to two rows, widening the frame. With scenarios enabled (×3 worlds) this grows further.
  • Asymmetry: the focus item may not be vertically centred — upstream might be 1 deep, downstream might be 4. The proportions shift as you re-focus.
  • Implementation: same CSS-flex approach as option A but rotated. No canvas/SVG engine needed; the vertical connectors are styled divs.

Good for wide upstream trees (many inputs); less comfortable than the river when the downstream is the bigger side.

Design notes — option C

  • Density: the most items per vertical pixel. No wasted connector space. Good for discovery ("what does K8 touch?") when the answer might be large.
  • Grouping: items grouped by depth AND edge type — more information per row than the graph layouts, since each row names its traversal edge.
  • Scale: handles Meridian's fan-out gracefully — the "+ N more" affordance is natural in list form. Deep chains just add more depth groups.
  • Loss: no visual sense of the graph structure — you can't see that K8 feeds a world that feeds verdicts at a glance. The topology is linearised into groups.
  • Implementation: the simplest of the three — pure HTML, no flex-lane geometry, no connector positioning.

Best for information density and large fan-out, but loses the spatial "graph" that the issue asks for. Consider as a secondary/collapsed view alongside A or B.

Placement — where does this view live?

  • Option 1: Full-screen overlay — triggered from the existing one-hop trace menu ("View full graph →" link at the bottom). Dimmed backdrop, close button, the graph fills the viewport. Lowest implementation cost; doesn't touch the tab structure.
  • Option 2: Observer tab sub-mode — a toggle on the Observer/bridge tab between the current staleness view and the dependency graph. The Observer role already holds the cross-cutting analysis surfaces (sensitivity, discrimination, staleness).
  • Option 3: Fifth tab — "Dependencies" alongside J-2 / Planner / Commander / Observer. Most visible, but adds permanent chrome to an already four-tab bar for a view that's used on-demand, not continuously.

Recommendation

  • Layout A (horizontal river) for the primary view — it matches the domain's left-to-right information flow and works at Meridian scale without a graph engine.
  • Layout C (layered list) as a compact fallback or tooltip detail for individual nodes — the grouping-by-edge-type is useful information that the river doesn't surface as clearly.
  • Placement 1 (full-screen overlay) from the trace menu — keeps the tab bar clean, makes the entry point the natural extension of the existing one-hop menu, and the overlay's viewport gives the river room to breathe.

All three layouts reuse TraceStore.walk + EDGE_ORIENTATION — the graph mechanics are already built. The remaining work is presentation only.
