How the composite works

  • River (A): gives the spatial overview — you see the shape of K8's influence at a glance. Upstream left, downstream right, each column one hop.
  • Sidebar (C): gives per-node detail — depth-grouped connections with edge types, metadata, and "+ N more" expansion. Opens on click, closes on ✕ or re-click.
  • Re-focus: double-clicking a sidebar node re-centres the river on it (not shown in this static mock, but the interaction model).
  • Sidebar width: fixed at 280px; collapses fully when dismissed. The river takes all remaining horizontal space.

Three options for where the dependency view lives in the app. Each mockup shows the same composite A+C view in its placement context. The inner content is identical — only the frame differs.

Strengths
  • Most discoverable — permanently visible in the tab bar
  • Full tab panel height — same room as other surfaces
  • Cross-tab glow would highlight the Dependencies tab when upstream knowledge changes
Trade-offs
  • Permanent chrome for an on-demand view — the four role tabs map to JIPOE roles; "Dependencies" is a tool, not a role
  • Five tabs may wrap on narrow viewports
  • Breaks the "four roles, one pipeline" conceptual model the app header announces

Recommendation

  • Primary: placement 1 (full-screen overlay) — the dependency view is an on-demand deep dive, not a persistent surface. The overlay gives it maximum room without adding permanent chrome.
  • Entry point: the existing one-hop trace menu gains a "View full graph →" link at the bottom. Click it → the overlay opens centred on that item.
  • Fallback consideration: placement 2 (Observer sub-mode) is a solid alternative if the modal nature of the overlay proves awkward in practice. It could be added later without conflict.
